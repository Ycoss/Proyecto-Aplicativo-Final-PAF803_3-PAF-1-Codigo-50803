Diapositiva 2: Introducción y Objetivos del Proyecto

Guion:

"Recreamos el clásico 'Asteroides': el jugador controla una nave, destruye asteroides y OVNIs, y evita colisiones para obtener la máxima puntuación."

"El objetivo principal fue aplicar la POO: diseño de clases, herencia y polimorfismo. Buscamos implementar un sistema de estados robusto, gestionar recursos eficientemente y desarrollar la lógica del juego, incluyendo controles modernos (WASD/Ratón), colisiones precisas (con láseres enemigos que dañan), y comportamientos de enemigos."

"Mejoramos la experiencia con niveles de dificultad (Fácil, Medio, Extremo) que cambian la velocidad de OVNIs y el color de la nave/láseres (azul, verde, rojo), una pantalla de Controles y funcionalidad de Pausa."

Diapositiva 3: Arquitectura General del Juego

Guion:

"Nuestra arquitectura por paquetes promueve la modularidad. main con la clase Window gestiona el bucle principal. gameObjects agrupa entidades como jugador, asteroides y OVNIs."

"Implementamos el patrón 'State' en el paquete states para manejar las fases del juego: carga, menú (con acceso a Controles y Dificultad), partida (con Pausa), y puntuaciones, facilitando las transiciones."

"Los paquetes graphics, input, io, ui y math proveen utilidades para gráficos (incluyendo texturas dinámicas por dificultad), entrada (teclado/ratón, barra espaciadora para Pausa), persistencia de datos, interfaz (botones para Controles) y matemáticas. La clase Constants en `gameObjects` es vital para la lógica de dificultad y cambios visuales."

Presentador 2

Diapositiva 4: Clases Creadas por el Programador (Parte 1 - Entidades del Juego y Estados)

Guion:

"Profundizando en las clases que desarrollamos, es importante reiterar que nos enfocaremos exclusivamente en aquellas creadas por nuestro equipo, omitiendo las clases predeterminadas de la API de Java."

"Comenzando con el paquete gameObjects, que define todas las entidades interactivas de nuestro juego:"

"Tenemos GameObject, nuestra clase base abstracta. Esta clase encapsula las propiedades fundamentales que todo objeto en nuestro juego debe tener, como una posición en el espacio bidimensional y una textura gráfica."

"De GameObject hereda MovingObject, otra clase abstracta, que añade la lógica y los atributos necesarios para el movimiento, como velocidad, ángulo de rotación, y la capacidad de detectar colisiones, incluyendo la lógica para diferenciar si un láser es del jugador o de un enemigo."

"Las clases concretas que heredan de MovingObject son: Player (nave con color dinámico por dificultad, dispara láseres no enemigos), Meteor (se dividen), Ufo (disparan láseres enemigos que dañan), Laser (proyectiles con color dinámico y bandera de enemigo) y PowerUp. Cada una especializa el comportamiento, como Player que responde a WASD/ratón."

"En states: State es la base abstracta para todos los estados. GameState, MenuState, ScoreState, LoadingState, ControlsState y PauseMenuState implementan State, manejando la lógica y UI de cada pantalla, incluyendo las nuevas pantallas de Controles y el menú de Pausa."

"Esta jerarquía y la herencia permiten reutilizar código y aplicar polimorfismo, como al manejar una lista de MovingObject."

Diapositiva 5: Clases Creadas por el Programador (Parte 2 - Soporte y Utilidades)

Guion:

"Desarrollamos clases de soporte y utilidad:"

"En graphics: Assets centraliza la carga de recursos visuales y auditivos (variantes de color para nave/láseres). Loader ayuda en la carga. Animation para efectos visuales. Text para dibujar texto. Sound maneja la reproducción y pausa/reanudación de audio."

"En input: KeyBoard captura teclas (WASD, flechas, barra espaciadora para Pausa). MouseInput para disparo y UI (botón 'Controles')."

"En ui: La clase Button y la interfaz Action para botones interactivos (botón 'Controles', botones de Pausa)."

"En io: JSONParser y ScoreData para persistencia de puntuaciones."

"En math, Vector2D. Window en main es el punto de entrada. Constants en gameObjects actualiza variables para el cambio visual dinámico (azul, verde, rojo)."

Diapositiva 6: Diagrama UML - Herencia e Implementaciones

Guion:

"Este diagrama UML muestra nuestra estructura de clases, destacando herencia (flecha triangular hueca) e implementaciones de interfaces (flecha discontinua)."

"Vemos herencia: GameObject es base de MovingObject, que a su vez es superclase de Player, Meteor, Ufo, Laser, PowerUp. Cada derivada hereda y especializa comportamiento. Similarmente, los estados (GameState, MenuState, ControlsState, PauseMenuState) heredan de State."

Diapositiva 7: Atributos y Métodos Relevantes (Clase Player y herencia de MovingObject)

Guion:

"La clase Player ejemplifica nuestra POO. Sus atributos clave (como heading, fireRate, shieldOn) son privados, mostrando encapsulamiento."

"Los métodos update(float dt) y draw(Graphics g), heredados de MovingObject, son sobrescritos en Player para su lógica específica: procesar entrada (WASD/flechas/ratón), moverse, rotar, disparar (creando láseres no enemigos), y dibujarse (seleccionando la textura de nave azul, verde o roja según dificultad)."

"El método Destroy(), también heredado y sobrescrito, maneja la pérdida de vidas y reaparición. Player tiene métodos propios como shootLaser(), que instancia láseres con el color correcto. Métodos como collidesWith() (que en MovingObject diferencia láseres enemigos) son heredados y utilizados."

Presentador 4

Diapositiva 8: Atributos y Métodos Relevantes (Clase GameState y uso polimórfico de MovingObject)

Guion:

"GameState maneja la lógica de la partida. Tiene una referencia al player y una ArrayList<MovingObject> llamada movingObjects para asteroides, OVNIs, láseres (con colores variables) y power-ups. Gestiona la entrada de pausa (barra espaciadora) que lleva a `PauseMenuState`."

"En los métodos update(float dt) y draw(Graphics g) de GameState, el polimorfismo es clave. Al recorrer movingObjects y llamar a mo.update(dt) o mo.draw(g), Java ejecuta la versión específica del método para cada tipo de objeto (Meteor, Ufo, etc.) cuando el juego no está pausado."

"Esta capacidad de tratar objetos de diferentes clases de forma uniforme pero obteniendo su comportamiento específico, simplifica el código de GameState y demuestra un uso eficiente del polimorfismo."
